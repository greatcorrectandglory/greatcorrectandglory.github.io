# 官网内容策略

## 页面定位

首页服务于客户的 30 秒理解：这是面向电子制造和研发验证的通用自动化测试平台，不是开发文档、源码说明或内部交付手册。独立页面服务 3 分钟场景评估和 10 分钟技术评估。首屏直接回答三个问题：

1. 场景：电子制造、研发验证、产线测试软件。
2. 能力：产品配置包、插件化测试步骤、多工位执行、硬件资源绑定、结果追溯、MES/云平台对接。
3. 价值：减少重复开发、缩短新产品导入时间、降低现场交付复杂度。

## 文案逻辑

页面从客户问题进入，而不是从技术栈进入：

- Hero 首屏：一句话定位平台和核心价值。
- 客户痛点：描述重复开发、设备接入不统一、追溯困难和系统集成复杂。
- 平台解决方案：把痛点映射到配置包、插件、资源绑定、结果追溯。
- 实际产品界面：没有脱敏真实截图前使用明确占位，避免伪造产品界面。
- 测试执行流程：先解释从选择产品到结果上报的工作方式，再进入技术架构。
- 核心架构图：独立架构页用公开抽象表达“配置 -> 运行时 -> 数据与集成”，不暴露内部源码结构。
- 三类用户入口：测试员、硬件工程师、软件工程师分别看到自己的使用价值。
- 典型应用场景：独立页面按“使用者、原有问题、平台如何介入、输出结果”说明研发验证、产线测试、烧录配置、多产品复用、质量追溯和 MES/云端对接。
- 核心功能清单：集中承载搜索关键词和能力完整性。
- 兼容性与交付边界：明确 Windows GUI、后台运行、配置格式、通信类型、结果存储、外部集成和插件类型。
- 部署与集成方式：说明静态官网不依赖后端，平台本体面向本地工站和企业系统。
- FAQ：处理开源边界、适用团队、与脚本的差异、静态部署问题。
- 联系/演示申请：引导通过 GitHub 联系，同时提醒不要公开提交敏感信息。

## SEO 关键词策略

首页关键词覆盖以下查询意图：

- 类目词：通用测试平台、自动化测试平台、产线测试软件、电子产品测试。
- 技术词：Python PyQt 测试平台、插件化测试系统、多工位测试。
- 集成和数据词：MES 对接、测试数据追溯、云平台对接。
- 对标词：OpenHTF、OpenTAP。

关键词放置原则：

- 每个页面只有一个 H1，且有唯一 `title` 和 `meta description`。
- 首页 `title` 同时包含平台名称、场景和测试软件属性。
- 首页 `meta description` 控制在 120 到 160 字，覆盖场景、能力和价值。
- 关键词自然分布在首屏、解决方案、功能清单和 FAQ 中，避免堆砌。
- Open Graph 使用同一套标题和描述，便于社交平台分享。
- `sitemap.xml` 明确列出首页、应用场景、技术架构、文档中心和版本记录。

## 信息安全边界

公开页面不写入以下内容：

- private 仓库源码、内部目录和实现细节。
- 客户名称、真实产品型号、工厂地址、网关地址、协议字段或公司内部系统信息。
- 产品配置包里的参数、真实设备标识、真实接口路径、账号或密钥。
- 未脱敏的截图、日志、序列号、工单号、MES 字段或真实设备地址。

如需展示案例，应使用匿名行业场景，例如“多工位产线测试”“研发样机验证”“新产品导入”，不要使用可反推客户或产品的信息。

## 公开文档策略

文档中心先发布目录和摘要，不直接搬运私有仓库文档。每篇正式文档发布前必须检查：

- 是否只描述当前已确认能力。
- 是否移除客户、内部协议、账号、密钥、真实设备地址和内部路径。
- 是否标注更新时间和适用版本。
- 是否把待确认能力写成“根据具体硬件项目评估”或“待确认”。
